# .PLAN-v1.md — Running Agenticana + GMI Side-by-Side in GitHub Actions

> **Version:** 1.0.0 | **Date:** 2026-03-13
> **Supersedes:** [.PLAN-v0.md](.PLAN-v0.md)
> **Goal:** Execute both **Agenticana** (20-agent AI system) and **GMI** (github-minimum-intelligence) as dual AI engines inside a single GitHub Actions workflow, sharing one installation folder and one workflow file.

---

## Decisions Made in v1

This plan incorporates three architectural decisions that reshape the execution model from v0:

| # | Decision | Impact |
|---|----------|--------|
| **D1** | Agenticana commands are prefixed with `?` to execute | `?` in issue title or comment body triggers the Agenticana engine |
| **D2** | All 20 agents run inside a single GitHub Action for up to 4 hours, with a 1-hour emergency override | `timeout-minutes: 240` on the agent job, plus a labeled emergency mode that extends to 300 minutes |
| **D3** | GMI actioned on `!`, Agenticana on `?` — both engines side-by-side in `.github-agenticana-intelligence/` folder with one workflow `.github-agenticana-intelligence-agent.yml` | Unified folder, unified workflow, dual-engine dispatch based on prefix character |

---

## Executive Summary

**v0** proposed adapting Agenticana to run as a solo agent engine in GitHub Actions, modeled after how github-minimum-intelligence (GMI) uses `pi`. **v1** goes further: both GMI and Agenticana coexist as dual engines inside a single repository folder (`.github-agenticana-intelligence/`) and are dispatched by a single workflow (`.github-agenticana-intelligence-agent.yml`). The prefix character in the issue title or comment body determines which engine handles the request:

- **`!`** → GMI engine (single generalist agent via `pi`)
- **`?`** → Agenticana engine (20 specialist agents with routing, skills, and ReasoningBank)

This means a repository owner installs **one folder** and **one workflow** to get both AI systems. They share state management, git commit patterns, authorization, and installation/upgrade infrastructure — but each engine has its own execution path, system prompt, and agent model.

## Architecture: Dual-Engine Dispatch

### The Closed Loop (v1)

```
User opens issue or comments with prefix:
    ? Build a user dashboard        → Agenticana engine
    ! Fix the typo in README        → GMI engine

    → GitHub Actions workflow triggers
    → Authorization check (collaborator with write+ access)
    → 🚀 reaction added as "working" indicator
    → Prefix detection:
        ? → Agenticana path (router → agent selection → multi-agent orchestration)
        ! → GMI path (single pi agent, direct execution)
    → Engine executes (up to 4 hours; 5 hours with emergency override)
    → Results posted as issue comment(s)
    → State committed to git
    → 👍 reaction on success, 👎 on error
```

### Prefix Routing Rules

| Prefix | Engine | Behavior |
|--------|--------|----------|
| `?` | **Agenticana** | Content is analyzed, routed to the best of 20 specialist agents. Multi-agent orchestration for complex tasks. ReasoningBank queried for past decisions. |
| `!` | **GMI** | Content is passed directly to the `pi` generalist agent. Single-agent, single-session, direct execution. |
| *(no prefix)* | **Ignored** | The workflow does not trigger on unprefixed content. This prevents accidental AI responses on regular issue discussions. |

### Prefix Detection Logic

```typescript
// In the unified orchestrator: .github-agenticana-intelligence/lifecycle/agent.ts

function detectEngine(content: string): "agenticana" | "gmi" | null {
  const trimmed = content.trim();
  if (trimmed.startsWith("?")) return "agenticana";
  if (trimmed.startsWith("!")) return "gmi";
  return null; // No prefix → do nothing
}

// For issue_comment events, check the comment body
// For issues.opened events, check the issue title first, then body
const input = event === "issue_comment"
  ? github.event.comment.body
  : github.event.issue.title || github.event.issue.body;

const engine = detectEngine(input);
if (!engine) {
  // Add 🤷 reaction and exit — no prefix, no action
  await addReaction(eventId, "confused");
  process.exit(0);
}

// Strip the prefix before passing to the engine
const prompt = input.trim().slice(1).trim();
```

### Key Structural Decisions

1. **Shared `state/`**: Both engines write to the same state directory. Issue-to-session mappings, transcripts, and audit logs are unified. This means you can `!` a quick fix and `?` a complex analysis on the same issue thread.

2. **Separate engine subdirectories**: GMI lives in `.pi/` (its native structure). Agenticana lives in `agenticana/` (agents, skills, router, reasoning-bank). The orchestrator (`lifecycle/agent.ts`) bridges both.

3. **Shared `install/`**: One installer handles both engines. Upgrades preserve user-edited files for both.

4. **Single `package.json`**: One dependency tree covering both `@mariozechner/pi-coding-agent` (for GMI) and any Agenticana runtime dependencies.

## The Workflow: `.github-agenticana-intelligence-agent.yml`

### Single Workflow, Three Jobs

File: `.github/workflows/.github-agenticana-intelligence-agent.yml`

```yaml
name: github-agenticana-intelligence-agent

on:
  issues:
    types: [opened]
  issue_comment:
    types: [created]
  push:
    branches: ["main"]
    paths-ignore:
      - ".github/workflows/**"
  workflow_dispatch:

permissions:
  contents: write
  issues: write
  actions: write
  pages: write
  id-token: write

jobs:
  # ─── Job 1: Self-Installer ────────────────────────────────────────────────
  run-install:
    runs-on: ubuntu-latest
    if: github.event_name == 'workflow_dispatch'
    steps:
      - uses: actions/checkout@v6

      - name: Install or Update
        run: |
          # Download .github-agenticana-intelligence/ from template repo
          # Handle fresh install vs upgrade:
          #   - Fresh: copy all files
          #   - Upgrade: backup user files (AGENTS.md, settings.json, state/)
          #              → install new version → restore user files
          # Handles both GMI (.pi/) and Agenticana (agenticana/) components
          # Semver comparison: install / upgrade / skip
          echo "Installing github-agenticana-intelligence..."

      - name: Commit installation
        run: |
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git add .github-agenticana-intelligence/ .github/workflows/.github-agenticana-intelligence-agent.yml
          git commit -m "chore: install/update github-agenticana-intelligence" || true
          git push

  # ─── Job 2: Dual-Engine Agent ─────────────────────────────────────────────
  run-agent:
    runs-on: ubuntu-latest
    timeout-minutes: 240  # 4-hour standard timeout for all 20 agents
    concurrency:
      group: agenticana-intelligence-${{ github.repository }}-issue-${{ github.event.issue.number }}
      cancel-in-progress: false
    if: >-
      (github.event_name == 'issues')
      || (github.event_name == 'issue_comment'
          && !endsWith(github.event.comment.user.login, '[bot]'))
    steps:
      - name: Authorize
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          REPO="${{ github.repository }}"
          ACTOR="${{ github.actor }}"
          PERM=$(gh api "repos/${REPO}/collaborators/${ACTOR}/permission" --jq '.permission')
          if [[ "$PERM" != "admin" && "$PERM" != "maintain" && "$PERM" != "write" ]]; then
            echo "❌ Unauthorized: ${ACTOR} has '${PERM}' permission (need write+)"
            exit 1
          fi
          echo "✅ Authorized: ${ACTOR} has '${PERM}' permission"

      - name: Detect prefix and engine
        id: detect
        env:
          EVENT_NAME: ${{ github.event_name }}
          ISSUE_TITLE: ${{ github.event.issue.title }}
          ISSUE_BODY: ${{ github.event.issue.body }}
          COMMENT_BODY: ${{ github.event.comment.body }}
        run: |
          # Determine the input text based on event type
          if [[ "$EVENT_NAME" == "issue_comment" ]]; then
            INPUT="$COMMENT_BODY"
          else
            INPUT="$ISSUE_TITLE"
            if [[ -z "$INPUT" ]]; then
              INPUT="$ISSUE_BODY"
            fi
          fi

          # Detect prefix
          TRIMMED=$(echo "$INPUT" | sed 's/^[[:space:]]*//')
          FIRST_CHAR="${TRIMMED:0:1}"

          if [[ "$FIRST_CHAR" == "?" ]]; then
            echo "engine=agenticana" >> "$GITHUB_OUTPUT"
            echo "prompt=${TRIMMED:1}" >> "$GITHUB_OUTPUT"
            echo "🧠 Agenticana engine selected (? prefix)"
          elif [[ "$FIRST_CHAR" == "!" ]]; then
            echo "engine=gmi" >> "$GITHUB_OUTPUT"
            echo "prompt=${TRIMMED:1}" >> "$GITHUB_OUTPUT"
            echo "⚡ GMI engine selected (! prefix)"
          else
            echo "engine=none" >> "$GITHUB_OUTPUT"
            echo "🤷 No prefix detected — skipping"
          fi

      - name: Exit if no prefix
        if: steps.detect.outputs.engine == 'none'
        run: |
          echo "No ? or ! prefix found. Exiting gracefully."
          exit 0

      - name: Add working reaction
        if: steps.detect.outputs.engine != 'none'
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          # Add 🚀 reaction to indicate processing
          if [[ "${{ github.event_name }}" == "issue_comment" ]]; then
            gh api "repos/${{ github.repository }}/issues/comments/${{ github.event.comment.id }}/reactions" \
              -f content=rocket
          else
            gh api "repos/${{ github.repository }}/issues/${{ github.event.issue.number }}/reactions" \
              -f content=rocket
          fi

      - name: Checkout
        if: steps.detect.outputs.engine != 'none'
        uses: actions/checkout@v6
        with:
          fetch-depth: 0

      - name: Setup Bun
        if: steps.detect.outputs.engine != 'none'
        uses: oven-sh/setup-bun@v2

      - name: Setup Node
        if: steps.detect.outputs.engine != 'none'
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Setup Python
        if: steps.detect.outputs.engine != 'none'
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'

      - name: Install dependencies
        if: steps.detect.outputs.engine != 'none'
        run: |
          cd .github-agenticana-intelligence && bun install --frozen-lockfile
          pip install sentence-transformers

      - name: Run engine
        if: steps.detect.outputs.engine != 'none'
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
          GEMINI_API_KEY: ${{ secrets.GEMINI_API_KEY }}
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          ENGINE: ${{ steps.detect.outputs.engine }}
        run: |
          bun .github-agenticana-intelligence/lifecycle/agent.ts

  # ─── Job 2b: Emergency Override (extends to 5 hours total) ────────────────
  run-agent-emergency:
    runs-on: ubuntu-latest
    timeout-minutes: 300  # 5 hours = 4h standard + 1h emergency
    concurrency:
      group: agenticana-intelligence-${{ github.repository }}-issue-${{ github.event.issue.number }}
      cancel-in-progress: false
    if: >-
      (github.event_name == 'issues' || github.event_name == 'issue_comment')
      && contains(github.event.issue.labels.*.name, 'emergency-override')
      && !endsWith(github.event.comment.user.login || '', '[bot]')
    steps:
      - name: Authorize
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          REPO="${{ github.repository }}"
          ACTOR="${{ github.actor }}"
          PERM=$(gh api "repos/${REPO}/collaborators/${ACTOR}/permission" --jq '.permission')
          if [[ "$PERM" != "admin" && "$PERM" != "maintain" ]]; then
            echo "❌ Emergency override requires admin or maintain permission"
            exit 1
          fi
          echo "✅ Emergency override authorized: ${ACTOR} has '${PERM}' permission"

      - name: Checkout
        uses: actions/checkout@v6
        with:
          fetch-depth: 0

      - name: Setup Bun
        uses: oven-sh/setup-bun@v2

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Setup Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'

      - name: Install dependencies
        run: |
          cd .github-agenticana-intelligence && bun install --frozen-lockfile
          pip install sentence-transformers

      - name: Run engine (emergency mode)
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
          GEMINI_API_KEY: ${{ secrets.GEMINI_API_KEY }}
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          ENGINE: agenticana
          EMERGENCY_OVERRIDE: "true"
        run: |
          echo "⚠️ EMERGENCY OVERRIDE: Extended timeout (5 hours)"
          bun .github-agenticana-intelligence/lifecycle/agent.ts

  # ─── Job 3: GitHub Pages ──────────────────────────────────────────────────
  run-gitpages:
    runs-on: ubuntu-latest
    if: github.event_name == 'push'
    permissions:
      pages: write
      id-token: write
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - uses: actions/checkout@v6
      - uses: actions/configure-pages@v5
      - uses: actions/upload-pages-artifact@v4
        with:
          path: '.github-agenticana-intelligence/public-fabric'
      - id: deployment
        uses: actions/deploy-pages@v4
```

### Timeout Model: 4 Hours + 1 Hour Emergency

| Mode | Timeout | Trigger | Authorization |
|------|---------|---------|---------------|
| **Standard** | 240 minutes (4 hours) | `?` or `!` prefix | Write+ collaborator |
| **Emergency** | 300 minutes (5 hours) | `emergency-override` label on issue | Admin or Maintain only |

**Why 4 hours?** All 20 Agenticana agents run sequentially inside a single workflow run. Complex multi-agent orchestration tasks — where the orchestrator dispatches to database-architect, then backend-specialist, then frontend-specialist, then test-engineer — can take significant time. The 4-hour window provides headroom for:

- Multi-agent orchestration with 4-6 sequential agents (~30-45 min each)
- Large codebase analysis tasks
- Full test suite generation and validation
- Complex architectural planning with Logic Simulacrum debates

**Why 1-hour emergency override?** Some tasks genuinely need more time (e.g., a full security audit of a large monorepo). The `emergency-override` label:

- Extends the timeout to 5 hours total
- Requires admin/maintain permission (higher bar than standard)
- Logs the override in the audit trail
- Is visible on the issue (the label serves as documentation)

// .github-agenticana-intelligence/lifecycle/gmi-engine.ts

export async function runGMI(prompt: string, event: GitHubEvent) {
  // 1. Resolve or create session mapping
  const sessionPath = await resolveSession(event.issue.number);

  // 2. Build pi system prompt from .pi/ directory
  const systemPrompt = await loadPiPrompt();

  // 3. Spawn pi agent with session
  const result = await spawnPi({
    prompt,
    session: sessionPath,
    systemPrompt,
    model: getConfiguredModel(),
  });

  // 4. Extract reply and post as comment
  await postComment(event.issue.number, result.reply);

  // 5. Persist session
  await persistSession(sessionPath);
}

#### 0.5 Implement Agenticana Engine (`agenticana-engine.ts`)

The Agenticana engine adds routing, multi-agent orchestration, and ReasoningBank:

```typescript
// .github-agenticana-intelligence/lifecycle/agenticana-engine.ts

export async function runAgenticana(prompt: string, event: GitHubEvent) {
  // 1. Resolve or create session mapping
  const sessionPath = await resolveSession(event.issue.number);

  // 2. Run the Router to determine agent + model tier
  const routerResult = await runRouter(prompt);
  // { agent: "frontend-specialist", model: "pro", tier: "MODERATE", skills: [...] }

  // 3. Query ReasoningBank for similar past decisions
  const bankResult = await queryReasoningBank(prompt);

  // 4. Build dynamic system prompt
  const systemPrompt = buildAgenticanaPrompt({
    agent: routerResult.agent,
    skills: routerResult.skills,
    reasoningBankContext: bankResult,
    geminiRules: await loadGeminiRules(),
  });

  // 5. Check if orchestrator is needed (multi-agent task)
  if (routerResult.agent === "orchestrator") {
    await runOrchestration(prompt, event, sessionPath, systemPrompt);
  } else {
    // Single agent execution
    const result = await spawnPi({
      prompt,
      session: sessionPath,
      systemPrompt,
      model: routerResult.model,
    });
    await postComment(event.issue.number, formatAgentReply(routerResult.agent, result.reply));
  }

  // 6. Record decision to ReasoningBank
  await recordDecision({
    task: prompt,
    agent: routerResult.agent,
    issueNumber: event.issue.number,
  });

  // 7. Persist session
  await persistSession(sessionPath);
}
```

### Phase 3 — Multi-Agent Orchestration (Swarm in a Single Action)

**Goal:** Enable the orchestrator agent to dispatch sub-tasks to specialist agents within a single 4-hour workflow run.

#### 3.1 Sequential Agent Invocation (Within the 4-Hour Window)

For complex multi-domain `?` tasks, the orchestrator dispatches agents sequentially **within the same Action run**:

```typescript
// orchestrator detects multi-domain task
const plan = await generatePlan(prompt);
// plan = [
//   { agent: "database-architect", task: "Design the schema", estMinutes: 30 },
//   { agent: "backend-specialist", task: "Build the API endpoints", estMinutes: 45 },
//   { agent: "frontend-specialist", task: "Create the UI components", estMinutes: 40 },
//   { agent: "test-engineer", task: "Write integration tests", estMinutes: 25 },
// ]

// Budget check: can we fit this in the remaining time?
const totalEstMinutes = plan.reduce((sum, step) => sum + step.estMinutes, 0);
const elapsedMinutes = getElapsedMinutes();
const remainingMinutes = (isEmergencyOverride ? 300 : 240) - elapsedMinutes;

if (totalEstMinutes > remainingMinutes * 0.8) {
  await postComment(issueNumber,
    `⚠️ Estimated ${totalEstMinutes}min needed, ${remainingMinutes}min remaining. ` +
    `Consider adding \`emergency-override\` label for extended time.`
  );
}

for (const step of plan) {
  const result = await runAgentStep(step.agent, step.task, sessionPath);
  await postComment(issueNumber, formatAgentReply(step.agent, result));
  results.push(result);
}
```

#### 3.2 Issue Threading

Each sub-task is posted as a separate comment, creating a visible execution thread:

```
Issue #42: "? Build a user dashboard with auth"
  └─ Comment 1: 🚀 Processing with Agenticana (? prefix detected)
  └─ Comment 2: 🤖 Orchestrator — Analyzing task, 4 agents needed (est. 140 min)...
  └─ Comment 3: 🗄️ database-architect — Schema designed: users, sessions, preferences
  └─ Comment 4: ⚙️ backend-specialist — API endpoints: /api/users, /api/sessions
  └─ Comment 5: 🎨 frontend-specialist — Components: Dashboard, UserProfile, AuthForm
  └─ Comment 6: 🧪 test-engineer — 23 tests written, all passing
  └─ Comment 7: ✅ Orchestrator — Task complete. Total time: 2h 18m
```

#### 3.3 Time Budget Management

Since all agents run inside a single Action (no matrix strategy), the orchestrator must manage time:

```typescript
class TimeBudget {
  private startTime: number;
  private maxMinutes: number;

  constructor(isEmergency: boolean) {
    this.startTime = Date.now();
    this.maxMinutes = isEmergency ? 300 : 240;
  }

  remainingMinutes(): number {
    const elapsed = (Date.now() - this.startTime) / 60000;
    return this.maxMinutes - elapsed;
  }

  canRunAgent(estimatedMinutes: number): boolean {
    // Leave 10-minute buffer for state commit and cleanup
    return this.remainingMinutes() > estimatedMinutes + 10;
  }

  isWarning(): boolean {
    return this.remainingMinutes() < 30;
  }
}
```

> **Note:** v0 proposed a matrix-based parallel dispatch strategy. v1 replaces this with sequential execution inside a single Action, which is simpler and avoids the overhead of multiple VM spin-ups. The 4-hour window provides enough time for even complex multi-agent tasks.

## Hybrid Model (Recommended)

The optimal setup runs all three modes simultaneously:

```
                    ┌──────────────────────────────────────────┐
                    │          GitHub Repository                │
                    │                                          │
  ┌─────────────┐  │  .github-agenticana-intelligence/        │  ┌──────────────┐
  │  VS Code    │  │  ├── agenticana/ (20 agents)             │  │  GitHub       │
  │  + Copilot  │──│  ├── .pi/ (GMI config)                   │──│  Issues       │
  │  (local)    │  │  ├── state/ (git-committed)              │  │  ? or !       │
  └─────────────┘  │  ├── lifecycle/agent.ts                  │  └──────────────┘
        │          │  └── public-fabric/                      │     │         │
        │          └──────────────────────────────────────────┘     │         │
        ▼                                                          ▼         ▼
  Local Copilot Chat                                         ? Agenticana  ! GMI
  (interactive dev)                                          (20 agents)   (1 agent)
        │                                                          │         │
        └──────────── Shared state via git pull/push ──────────────┘─────────┘
```

- **Local mode:** Developer uses Copilot Chat with MCP tools for interactive work.
- **`?` mode:** Team members open Issues with `?` for Agenticana's 20-agent intelligence (complex tasks, multi-agent orchestration, ReasoningBank).
- **`!` mode:** Team members open Issues with `!` for GMI's quick single-agent responses (simple tasks, direct execution).
- **All three share:** The same agents, skills, ReasoningBank, and state via git.

## Open Questions (Updated from v0)

1. **Which LLM provider should be the default?** Support both Gemini and OpenAI. Default to configurable via `.pi/settings.json`.
2. **Should `?` without an agent label always auto-route, or should it default to a "general" agent?** Recommendation: auto-route via complexity scorer.
3. **How should the 4-hour budget be split across multi-agent orchestration steps?** Fixed allocation per agent, or dynamic based on task complexity?
4. **Should `!` GMI responses also be recorded in the ReasoningBank?** Currently planned as Agenticana-only. Could be extended.
5. **What happens when a `?` task hits the 4-hour limit mid-execution?** Save partial state, post a summary of completed work, and suggest the user add `emergency-override` label?
6. **Should the emergency override be automatic (label-based) or require explicit re-trigger?** Currently: separate job with higher timeout. Alternative: single job that checks for label and adjusts timeout dynamically.
7. **Prefix collision: what if a user genuinely starts a comment with `?` or `!` without intending to trigger the engine?** Could require a space after the prefix (e.g., `? ` not `?word`) or a double-prefix (`??`, `!!`).
